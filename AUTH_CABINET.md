# Личный кабинет и авторизация

Документ ведётся по ходу разработки: архитектура, переменные окружения, API и эксплуатация.

## Цели

- Открытая регистрация с подтверждением email и восстановлением пароля.
- Роли: `user` и `admin`. Первый зарегистрированный пользователь автоматически получает `admin`; далее роль назначает администратор.
- Весь сайт доступен только авторизованным пользователям; HTTP Basic Auth удалён.
- Отдельные URL и отдельный интерфейс кабинета (русский язык).
- API поиска — только с персональным API-ключом пользователя.
- PostgreSQL на проде; резервное копирование БД сразу; миграции — позже (пока `schema/init.sql` + инициализация при старте).

## Роли и возможности

### Пользователь (`user`)

- Поиск (интерфейс в кабинете), история запросов.
- Лимиты на использование (дневные/месячные, размер пакета).
- Баланс (учётная запись; списание — по правилам сервиса).
- Собственные ключи провайдеров LLM (перекрывают или дополняют серверные `.env`).
- Персональные API-ключи для вызова `POST /api/query` и `POST /api/query/stream`.

### Администратор (`admin`)

- Всё из роли пользователя.
- Управление пользователями: список, блокировка, смена роли, лимиты, баланс.
- Просмотр агрегированной статистики и аудита (события безопасности пишутся в файл логов).

## Аутентификация

- Вход: email + пароль.
- **Access JWT** (короткий срок) в заголовке `Authorization: Bearer` для API и SPA.
- **Refresh-токен** в httpOnly cookie для обновления сессии в браузере (отдельно от API-ключей для интеграций).
- Подтверждение email и сброс пароля — одноразовые токены в БД, ссылка в письме.
- API-ключи: префикс + секрет; в БД хранится только хеш; заголовок `X-API-Key` или `Authorization: Bearer <ключ>`.

## Маршруты (черновик)

| Путь | Назначение |
|------|------------|
| `/login`, `/register`, `/forgot-password`, `/reset-password`, `/verify-email` | Публичные страницы auth |
| `/cabinet/` | Кабинет пользователя |
| `/cabinet/search` | Поиск (перенос текущего UI) |
| `/admin/` | Панель администратора |
| `/api/auth/*` | Регистрация, вход, refresh, выход, verify, reset |
| `/api/cabinet/*` | Профиль, ключи провайдеров, API-ключи, история, баланс |
| `/api/admin/*` | Управление пользователями и настройками |
| `/api/query`, `/api/query/stream`, `/api/meta` | Только с API-ключом пользователя |

## Переменные окружения

См. `.env.example` (блок `PostgreSQL`, `JWT`, `SMTP`, `APP_ENCRYPTION_KEY`).

## База данных

- Схема: `schema/init.sql`.
- Инициализация при старте приложения, если таблиц ещё нет.
- Бэкап: `scripts/backup-postgres.sh`.

## Аудит

События входа, смены роли, выдачи/отзыва API-ключей и т.п. — в JSONL (`logs/audit.log` или общий `LOG_FILE` с полем `audit: true`).

## Статус реализации

- Реализованы: регистрация, подтверждение email, сброс пароля, JWT + refresh-cookie, кабинет, админка, API-ключи, баланс/лимиты, история, пользовательские ключи провайдеров, бэкап `scripts/backup-postgres.sh`.
- Поиск в UI кабинета: `POST /api/cabinet/query` и `POST /api/cabinet/query/stream` (Bearer access JWT).
- Внешний API: `POST /api/query`, `POST /api/query/stream`, `GET /api/meta` — персональный API-ключ.
- HTTP Basic Auth и `API_SECRET` как основная защита сняты; переменные `SITE_BASIC_AUTH_*` и `API_SECRET` в `.env.example` оставлены только как историческая справка и не используются приложением.
